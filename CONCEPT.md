# コンセプトドキュメント

> バイク車種別の通行制限を考慮したGoogleマップ連携アプリ
> 壁打ち時点での合意・未決事項のスナップショット

## 1. プロジェクトの目的

排気量によって通れない道路があるバイクライダーのために、車種に応じた通行制限を考慮したルート案内を提供する。Googleマップは車種別の通行制限を考慮しないため、原付や125ccライダーが知らずに自動車専用道路に誘導されるなどの不便がある。これを解消する。

## 2. 対象ユーザー

- 原付一種(50cc相当・新基準原付): 自動車専用道路NG、高速道路NG、二段階右折義務
- 原付二種(125cc以下): 自動車専用道路NG、高速道路NG
- 普通二輪以上(126cc〜): 全道路OK、高速道路で二人乗りは年齢・経験条件あり

主な利用シーンは「出発前の下調べ」(自宅PCやスマホでルート確認 → ツーリング当日はGoogleマップで案内)。ツーリング中のリアルタイム利用は当面想定しない。

## 3. プロジェクトのポジショニング

商用の「ツーリングサポーター by NAVITIME」は、警察提供データと自社整備の道路ネットワークを使った独自ルーティングエンジンで車種別案内を実現している。これは個人開発で再現できる規模ではない(20年以上の蓄積、警察との関係、月額有料モデルが前提)。

ナビタイムに対する優位は「無料・OSS・コミュニティ駆動」であること。Googleマップに対する優位は「車種別の通行制限を考慮できる」こと。完璧でなくてもGoogleマップよりマシなら価値がある、というニッチを狙う。

## 4. 議論の経緯と意思決定の記録

### 4.1 形態とスコープ

| 論点 | 決定 |
|---|---|
| アプリ形態 | Webアプリ(下調べ中心の利用シーンに合わせる、ネイティブ化は当面なし) |
| 利用者 | 公開してライダー全般に |
| 公開戦略 | まず作る、公開は後で考える |
| 運営体制 | 最初からOSS化してコミュニティと作る |

「スマホネイティブアプリ」が当初の希望だったが、利用シーンが「出発前の下調べ」と決まった時点で、ネイティブの強みであるバックグラウンド動作・常時位置取得・オフライン地図が不要になり、Web一択と再判断した。

### 4.2 車種区分

3区分(50cc / 125cc / 126cc以上)に決定。シンプルさ重視。

実態に即した正確な呼称は「原付一種 / 原付二種 / 普通二輪以上」だが、UI上は親しみやすい排気量表記を採用予定。2025年4月から「新基準原付」(125ccエンジン出力制限)が原付一種扱いになっているため、内部では「区分」を保持しつつ表示は柔軟にする。

### 4.3 車種選択とavoidオプション

「車種選んだらおまかせ」方式に決定。avoid設定は車種ごとに固定し、ユーザーには見せない。

| 車種 | 内部設定 |
|---|---|
| 50cc | avoid=highways,tolls,ferries |
| 125cc | avoid=highways,tolls |
| 126cc以上 | avoidなし |

これに加えて、フェーズ1以降で「禁止区間DB」と空間照合し、Googleマップのavoidパラメータでは表現できない車種別規制を補完する。

### 4.4 データモデルの方針

| 論点 | 決定 |
|---|---|
| 投稿の粒度 | 区間方式(始点・終点+道路名) |
| 投稿の信頼性担保 | 写真必須+複数の確認票で確定 |

ポイント方式(1点)では精度が低く、ルート方式(OSMリンクID)では一般ユーザーに難しい。区間方式が現実解。

### 4.5 ルーティングのアプローチ

「自前ルーティング(NAVITIME相当)」は最初の希望だったが、調査の結果、個人開発・OSSでは到達不可能と判断し撤回。代替として、**Googleマップ + ライダー投稿の禁止区間DB のハイブリッド**に方針変更した。

ルーティング自体はGoogle(URLs API or Routes API)に任せ、その結果のpolylineと禁止区間DBを空間交差判定して警告を出す方式。これにより:

- 高品質なルーティングはGoogleの資産を借りる
- 車種別の規制情報だけ自前で持つ(コスト最小化)
- 完璧ではないが「Googleマップよりマシ」を確実に達成できる

### 4.6 データソースの3層構造

ナビタイム調査と日本二輪車普及安全協会(JMPSA)サイトの存在を踏まえて、データソースを3層に整理:

**レイヤー1: JMPSAデータ(全車種共通禁止)**
- 全国約500箇所、警察提供の公式情報
- 騒音・事故対策で二輪車一律禁止の区間が中心
- **JMPSAに確認済み: 非商用利用であれば利用可**(2026年5月時点)
- この条件は、プロジェクト全体に大きな制約となる。詳細は後述のライセンス・制約方針に反映する

**レイヤー2: 公的オープンデータ(車種別)**
- 国土数値情報の道路データから自動車専用道路を特定
- OpenStreetMapの `motorroad=yes` タグも補完的に利用
- 125cc以下NG区間の特定に有用

**レイヤー3: ユーザー投稿(粒度の細かい補完)**
- レイヤー1・2に漏れた地方バイパス・細かい規制を埋める
- 写真付き投稿+承認制で信頼性担保
- コミュニティの参加感も生む

## 5. 未決定の論点

ここから先は手を動かしながら詰める領域。机上の議論を続けても精度は上がらない。

- 国土数値情報の自動車専用道路データの実用性(実検証が必要)
- ライセンス選定: **非商用制約があるためMITは不適**。代替案として:
  - Creative Commons BY-NC 4.0(非商用限定)をコード+データに適用
  - PolyForm Noncommercial License 1.0.0(コードのみ非商用ライセンス)
  - 独自ライセンス(JMPSAデータは非商用利用禁止、コード自体はMITと分離)
- 投稿データのライセンス(CC BY-NC 4.0 が筆頭候補)
- マネタイズ: **広告・課金は商用扱いになる可能性が高く不可**。代替は寄付(Buy Me a Coffee, GitHub Sponsors)・運営者負担・助成金等
- サーバー代の負担方針(個人負担 / 寄付ベース / OSSスポンサー)
- 利用規約・プライバシーポリシーの整備
- ローンチ戦略

### 非商用利用制約の含意

JMPSAデータを利用する以上、このプロジェクトは**非商用プロジェクトとして運営する**ことが前提となる。これは:

- アプリ自体に広告を載せない
- 課金機能・サブスクリプションを設けない
- 派生プロジェクトも非商用に限る(ライセンスで継承)

ただし、将来本格的に有料展開したい場合に備えて:

- JMPSAデータを取り外せる設計にしておく(レイヤー1を抜いても動く構造)
- 自前データ(ユーザー投稿レイヤー)を強化して代替できるようにする

データソースの3層構造は、この「将来の選択肢」を残す意味でも有効。

## 6. リスクと対応方針

**法的リスク**: 「アプリ案内通りに走ったら違反した」というクレーム。対応は免責事項の明示+「最終的な通行可否は道路標識を確認すること」を全画面で表示。

**データの正確性**: 投稿型は誤情報リスクあり。対応は写真必須+複数確認票による承認制。公式ソース(JMPSA・警察告示)とのリンクを推奨。

**コールドスタート**: データが少ないと使えないアプリと評価される。対応はJMPSAデータ500件を初期投入(利用可否確認後)+主要バイパス50件は手動投入。

**運営者個人の負担**: 投稿審査・モデレーションが個人の手に余る可能性。対応はOSS化により協力者を募る。
